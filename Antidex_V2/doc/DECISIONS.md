# Decisions — Antidex_V2
- 2026-02-15: Cree le sous-projet `Local_Agents/Antidex_V2/` et demarre par la doc (SPEC/TODO/TESTING_PLAN + index) selon `Local_Agents/doc/DOCS_RULES.md`.
- 2026-03-14: La future compaction/archivage du contexte ne doit pas etre une intelligence cachee de l'orchestrateur. Direction retenue: approche **pilotee par les agents**, surtout le Manager, avec `context_checkpoint.md` comme resume canonique de travail, `archive/` comme historique consultable non directif, et de possibles helpers explicites plus tard. Rationale: garder Antidex leger cote cadre, tout en reduisant la pollution du contexte actif sur les taches longues ou tres iteratives.
- 2026-03-14: Sur les taches outcome-driven en REWORK, la regle de fraicheur des preuves peut pousser le Developer a rerunner un benchmark identique trop tot. Direction retenue: ajout d'un **opt-in explicite du Manager** (`Reviewed evidence may be reused for planning this step: yes|no`) dans `manager_instruction.md` / `manager_review.md`, lu par le prompt Developer. Cette autorisation sert uniquement a choisir la prochaine modification; elle ne supprime pas l'exigence de preuve fraiche avant `ready_for_review`.

- 2026-03-09: Long jobs Windows: la forme recommandee devient `--script` ou argv apres `--`, persistee en `command_argv` puis lancee sans shell quand possible. `--command` reste compatible mais est relegue au mode legacy car il casse facilement avec du quoting imbrique sous Windows.
- 2026-03-09: TODO editable: `Save + Continue` reste supporte, mais la detection `todo_updated` devient dedupee par fingerprint de contenu (`lastReconciledTodoFingerprint`) pour eviter de retraiter le meme TODO deja reconcilie apres un simple re-save/restart.
- 2026-03-09: Review Manager outcome-driven: le prompt doit fournir un template explicite `manager_review.md` avec labels obligatoires (`Goal check`, `Rerun justification`, etc.) afin de reduire les retries de postconditions. Tant qu'un tour Manager est actif, le statut canonique du run reste cote Manager pour eviter les faux signaux de dispatch developer.
- 2026-03-09: Monitor long-job: le premier rapport attend au moins 5 minutes par defaut, et toute decision destructive est differee pendant une fenetre de warmup silencieux si le pid est encore vivant et qu'aucun artefact d'echec n'existe.
- 2026-03-09: Long jobs Windows: ne pas encapsuler un outil a flags (ex: `tsx src/runner.ts`) derriere un `npm run ...` fragile sans verifier le passage des arguments. Dans `ai_lab`, cela transformait `--players 2 --games 200 --output ...` en positionnels (`2 200 easy 200 ...`) et ecrivait un fichier parasite `200`. Decision: preferer un wrapper JS protocolaire qui lance l'outil final directement et ecrit `heartbeat/progress/result`.
- 2026-03-09: Long jobs: un `result.json` terminal explicite en echec (`error|failed|stopped|canceled`) n'est pas un crash brut. L'orchestrateur doit reveiller le developer avec le dernier diagnostic visible, et reserver la classification `crashed` au cas "pid mort sans resultat terminal exploitable".
- 2026-02-15: Decision de nommage: le projet s'appelle "Antidex_V2"; la doc canonique est `doc/SPEC.md`.
- 2026-02-15: Mapping des anciennes consignes "todo/" vers les regles actuelles: `doc/TODO.md` (et docs par feature dans `doc/`), plutot qu'un dossier `todo/` separe.
- 2026-02-15: Communication entre agents: protocole 100% fichiers avec `data/pipeline_state.json` + 1 dossier par tache (`data/tasks/T-xxx_<slug>/...`) + mailboxes "to/from" (`data/mailbox/...`) + separation ACK (B) vs RESULT (C).
- 2026-02-15: Politique threads: Manager (Codex) conserve un thread unique (toujours resume). Les deux developpeurs (Codex + Antigravity) suivent la meme politique: le Manager decide, par defaut reuse; bascule en `new_per_task` uniquement pour gros projets ou degradation.
- 2026-02-15: Systemes d'instructions agents: `agents/<role>.md` (Base + Overrides, version/update_at) + header "READ FIRST" injecte par l'orchestrateur au debut de chaque prompt pour forcer la lecture et pointer les chemins a lire/ecrire.
- 2026-02-15: Auto-correction: le Manager peut modifier les instructions des agents (`agents/*.md`) y compris la partie "Base" si necessaire pour corriger un dysfonctionnement; obligation de tracer le changement dans `doc/DECISIONS.md` et d'incrementer la `version`.
- 2026-02-16: Bootstrap: l'orchestrateur cree un squelette **uniquement** dans le projet cible (`cwd`) au demarrage d'un run (doc/ + agents/ + data/), de maniere non destructive.
- 2026-02-24: Robustesse anti-boucle: `manager_decision` est traite comme un marqueur **one-shot** (consomme puis remis a `null` par l'orchestrateur). Postconditions renforcees sur le tour "Manager review": ACCEPTED implique d'avancer `current_task_id` + `developer_status=ongoing` (sinon blocage avec question actionnable). Ajout d'un garde-fou "review loop" (limite d'essais) pour eviter la brulure de tokens.
- 2026-02-27: Le prompt developer inclut `manager_review.md` quand il existe pour exposer le feedback REWORK et reduire les boucles de re-dispatch.
- 2026-02-28: AG prompt fournit aussi des chemins absolus + l'orchestrateur tolere l'ACK manquant si RESULT valide, pour reduire les stalls AG.
- 2026-02-28: Watchdog AG: seuil de stall distinct pour la phase waiting_result (ACK observe) via ANTIDEX_AG_STALL_RESULT_MS pour reduire les faux positifs sur taches longues.
- 2026-02-28: Watchdog AG: ACK best-effort (activity implies delivery) + auto-resend once on first ACK timeout before handoff Manager.
- 2026-03-01: Review Manager: si ACCEPTED + continue vers une nouvelle tache, l'orchestrateur exige que `task.md` + `manager_instruction.md` de la tache suivante existent (postconditions + prompt) pour eviter les blocages "missing task spec".
- 2026-03-01: Watchdog AG: ajout d'un override par tache (`ag_expected_silence_ms` / `ag_expected_silence_minutes`) lu depuis task.md / manager_instruction.md pour limiter les faux stalls sur longues phases browser-only.
- 2026-03-02: Guardrail loop: reponse a `Q-loop` doit changer l'etat (developer_status=ongoing/ready_for_review ou manager_decision=blocked/completed) pour casser les boucles auto-run.
- 2026-03-03: Override utilisateur "Send to manager": accepter le queueing meme pendant un run actif et traiter le user_command en priorite avant tout dispatch/review suivant.
- 2026-03-04: Robustesse "Send to manager" post-completion: postcondition renforcee sur `manager/user_command` (doit produire un NEXT task via TODO non-coché, ou `manager_decision=completed`) pour eviter le "re-complete" silencieux.
- 2026-03-04: Corrector: `manager/user_command` (postconditions failed apres retries) est traite comme un guardrail pour permettre un incident + intervention Correcteur (quand active) sur les soucis Antidex (prompts/UX), pas sur le projet cible.
- 2026-03-04: Corrector robustness: si `threadResume` echoue avec une erreur RPC `-32600` / "no rollout found", demarrer un nouveau thread Corrector (fallback) au lieu d'echouer.
- 2026-03-04: Quota-aware dispatch UX: quand le quota AG est <= 40%, le prompt Manager inclut un rappel explicite qu'il peut devoir changer l'assignation des taches (doc/TODO.md) pour alterner AG/Codex.
- 2026-03-04: UX monitoring: reduire les re-renders/polling UI quand SSE est actif (eviter l'effet "rafraichissement" dans le navigateur), garder un backstop polling lent si SSE est stale.
- 2026-03-03: Robustesse encodage: reparer automatiquement le mojibake (UTF-8 interprete latin1) dans le state store au chargement pour que les chemins (TODO/logs) restent valides.
- 2026-03-05: Dispatch AG: apres 3 stalls watchdog, bloquer via question "AG disabled" avant tout guardrail `dispatch_loop` pour forcer une decision Manager (switch dev ou override explicite).
- 2026-03-05: Dispatch AG: appliquer ce guardrail "AG disabled" directement dans `_bumpDispatchCountOrBlock` pour garantir la precedence quel que soit le chemin d'appel.
- 2026-03-05: Incidents auto-run: si `dispatch_loop` concerne AG avec 3 stalls, reconcilier vers "AG disabled" et ne pas declencher Corrector (guardrail = action Manager).
- 2026-03-05: Dispatch AG (REWORK): si `manager_review.md` existe et que la decision est REWORK, inclure explicitement ce fichier dans le prompt AG (et marquer le dispatch comme REWORK) pour eviter les boucles "rework non lu".
- 2026-03-05: Continue recovery: apres `Stop` puis `Continue pipeline`, reset les compteurs transients (retry/dispatch) pour la tache courante afin de re-tenter proprement apres correction d'un probleme externe (connector/AG).
- 2026-03-05: Corrector sans supervisor: declencher quand meme le Corrector (pour produire le correctif), mais sans auto-restart. Apres fix, stopper le run et signaler "restart requis" (eviter les boucles d'incidents).
- 2026-03-06: Antidex_V2: ajouter un mode "Corrector externe" pilote par un Guardian (daemon). Sur incident, Antidex ecrit un marker `data/external_corrector/pending.json` et stoppe le run; le Guardian declenche ensuite `POST /api/corrector/run_pending`, et gere les restarts (code 42) + Continue best-effort. Rationale: isolation process + reprise plus robuste que le Corrector in-process seul.
- 2026-03-06: Guardian: permettre les retries sur un meme marker pending (cooldown), et traiter un echec `run_pending` avec marker disparu comme "handled/restart in progress" (evite le spam si le serveur redemarre pendant la requete).
- 2026-03-06: TODO rebase sans spec: traiter l'absence de `task.md`/`manager_instruction.md` comme une action Manager attendue (blocage + Q-missing-task-spec) et **ne pas** declencher le Corrector; utiliser un marqueur `lastError.source=todo_rebase` pour court-circuiter l'auto-fix.
- 2026-03-06: Turn inactivity: suspendre le timeout d'inactivite quand une `commandExecution` longue est en cours; s'appuyer sur le hard-timeout pour eviter les runs infinis.
- 2026-03-06: Review loop: reinitialiser le compteur de reviews apres un review valide (state coherents) pour eviter un guardrail apres plusieurs cycles legitimes.
- 2026-03-06: Review loop: apres reponse Manager a `Q-review-loop`, reset du compteur de reviews; reprise auto apres REWORK basee sur `dev_result.*`/result.json (pas `dev_ack.json`) pour eviter les faux "nouveaux resultats".
- 2026-03-06: Dispatch loop: apres un review valide, reset du compteur de dispatch pour la tache afin d'eviter le guardrail `dispatch_loop` sur des cycles de rework deja reviews.
- 2026-03-08: Corrector: traiter "Run stopped" comme un stop utilisateur et ne jamais declencher le Corrector, meme si `lastError.where=manager/user_command` ou `auto`.
- 2026-03-08: AG watchdog: `ag/watchdog` est une action Manager (Q-watchdog) et ne declenche pas le Corrector.
- 2026-03-08: Long jobs: `job/crash` tente 1 auto-restart puis bloque le Manager avec `Q-job-crash`; le Corrector ne se declenche pas automatiquement.
- 2026-03-08: User command queue V2: remplacer le slot unique `pendingUserCommand` par un modele `pending + queued bundle`. Un 2e `Send to manager` pendant un override cree un `queuedUserCommand`; les messages suivants sont fusionnes dans ce bundle et doivent etre livres au Manager avant tout dispatch developer, afin d'eviter toute perte de message utilisateur.
- 2026-03-08: Tooling Playwright: `npm run test:pw` passe par `scripts/run-playwright.js`, qui resout d'abord le Playwright local puis un fallback compatible dans `../Antidex/node_modules`, afin d'eviter les echecs lies a `npx` quand `Antidex_V2/node_modules` est absent.
- 2026-03-08: Long jobs / stop semantics: un run `stopped` (en particulier `corrector/restart_required`) doit geler l'etat orchestrateur. On nettoie `activeJobId/activeJob`, on reecrit le `developer_status` projet en `idle`, et la boucle `_tickLongJobs()` ignore les runs terminaux avant toute adoption/restart afin d'eviter `stopped + waiting_job` et les jobs fantomes dans l'UI.
- 2026-03-08: UI/API long jobs: le panneau doit afficher le dernier job connu meme sans `activeJob` (fallback `lastJobId`), et le reveil developer doit utiliser une formulation outcome-aware (`crashed`/`completed`/`stopped`/`ended`) pour eviter les faux messages `Long job finished`.
- 2026-03-07: Les calculs longs `developer_codex` doivent sortir du modele "tour LLM ouvert" et passer par un mode "long job" explicite. Antidex fournit l'enveloppe reusable (launch background, etats, UI, monitor, watchdog, correcteurs); le code metier du calcul reste a ecrire dans le projet cible par le developer.
- 2026-03-07: Le mode long job V1 est limite a `developer_codex`. Un monitor Codex horaire ecrit un rapport visible dans l'UI, peut stopper/restart dans un perimetre borne, et l'absence de rapport monitor a l'heure devient un incident prioritaire pour le Correcteur externe.
- 2026-03-07: Implementation V1: demarrage via requete fichier `data/jobs/requests/*.json` (helper projet `tools/antidex.cmd job start ...`), supervision dans l'orchestrateur (`waiting_job`, reprise apres restart), UI + API `/api/jobs/*`, monitor reports `monitor_reports/latest.*` + `REP-*`.
- 2026-03-07: Handshake Codex: traiter **marker+postconditions** comme succes; si `turn/completed` n'arrive pas apres un court delai, detacher le tour et force-unblock pour eviter le wedge ("Another turn is already running"). Tracer `turn_detached_after_marker`.
- 2026-03-07: Manager review: chaque tour "reviewing" doit modifier `data/tasks/<task>/manager_review.md` (mtime change) pour eviter les boucles sur un review stale (ex: "1 seul REWORK" alors que le run boucle).
- 2026-03-10: Manager review freshness: ne plus dependre uniquement du `mtime` de `manager_review.md`; le prompt impose `Turn nonce: <turn_nonce>` et l'orchestrateur valide en priorite ce nonce de tour pour eviter les faux stale reviews.
- 2026-03-10: REWORK same-task rebase: lors d'un `manager_decision=continue` sur la tache courante, la rebase TODO doit preserver `developer_status=ongoing` et le summary Manager au lieu de re-promouvoir `ready_for_review` a cause d'un ancien `dev_result.*`. Sinon le run boucle en review sans nouveau dispatch developer.
- 2026-03-10: REWORK outcome-driven stale evidence: sur une tache outcome-driven en `Decision: REWORK`, ni le Developer ni le Manager answering ne peuvent remettre `developer_status=ready_for_review` sur la seule base d'un `dev_result.*` re-ecrit. Il faut des artefacts de preuve frais (reports/result.json references par `task.md` / `manager_instruction.md`) plus recents que `manager_review.md`.
- 2026-03-11: REWORK outcome-driven cited-proof freshness: si `dev_result.md|json` cite explicitement un report `reports/*.json`, ce report fait partie de la preuve de review. Son `meta.generated_at`/freshness doit etre plus recent que `manager_review.md`; un `dev_result.*` plus recent ne peut pas masquer un report stale qu'il cite encore.
- 2026-03-10: `waiting_job` stale recovery: un `Continue` sur `developer_status=waiting_job` sans job vivant et sans requete pending doit revenir a `implementing/ongoing` au lieu de rester bloque indefiniment.
- 2026-03-11: `waiting_job` + terminal result reconciliation: si le dernier long job a deja ecrit un `result.json` terminal mais que `data/pipeline_state.json` reste en `developer_status=waiting_job` (ex: stop manuel / cleanup tardif), l'orchestrateur doit reecrire le fichier en `developer_status=ongoing` et reveiller logiquement le developer pour interpreter le resultat existant, sans relancer immediatement un nouveau job.
- 2026-03-11: Auto-promotion `waiting_job`: ne jamais inferer `developer_status=waiting_job` a partir d'un vieux `job.json` encore marque `running` si le job a deja un `result.json` terminal. L'auto-promotion vers `waiting_job` exige maintenant une requete pending protocol-aware ou un vrai job vivant sans resultat terminal.
- 2026-03-11: Auto-promotion `ready_for_review`: l'orchestrateur doit reappliquer les guardrails outcome-driven lors de l'auto-promotion `ongoing -> ready_for_review`. Sans cela, une tache `REWORK` peut reboucler si le developer reecrit seulement `pipeline_state.json`/`dev_result.*` sans regenerer les preuves demandees.
- 2026-03-11: La trace brute des long jobs (`data/jobs/<job_id>/...`) est conservee, mais Antidex ajoute une **memoire consolidee par tache** (`data/tasks/<task_id>/long_job_history.json|md`) comme vue canonique de decision. Cette memoire est regeneree par l'orchestrateur, alimentee par les jobs/resultats/reviews, lue par les prompts Manager/Developer, et exposee aussi dans l'API/UI jobs.
- 2026-03-12: Handoff post-long-job: la memoire consolidee `long_job_history.*` ne suffit pas juste apres `wake_developer`, car les agents relisent aussi des docs manager stale. Decision: ajouter un artefact systeme canonique `data/tasks/<task>/latest_long_job_outcome.json|md`, regenere sur resultat terminal, lu avant `manager_instruction.md` / `manager_review.md`, et utiliser aussi ce moment pour nettoyer `pipeline_state.json.summary/tests.notes` vers "terminal result to consume now".
- 2026-03-12: Le handoff post-long-job doit parser le `result.json` tel qu'il existe vraiment dans les wrappers projet (`output` + `summary` au niveau racine, pas seulement `outputs[]`). Le resume packet developer doit aussi etre ecrit apres reconciliation terminale, sinon il peut figer un faux `waiting_job` et contredire `latest_long_job_outcome.md`.
- 2026-03-12: Un `developer_status=blocked` normal apres consommation d'un long-job terminal n'est pas un incident. Decision: si le developer ecrit une question `Q-*.md` et passe `developer_status=blocked`, Antidex doit router immediatement vers `manager/answering` au lieu de declencher le Correcteur.
- 2026-03-11: Long-job display state: pour l'API/UI, un `result.json` terminal prime sur un `job.json` stale. Un dernier job avec pid mort + `result.json.status=done|error|failed|stopped` ne doit plus etre affiche `running`.
- 2026-03-11: Reconciliation terminal long-job: si un `result.json` terminal existe deja mais que `job.json` / `monitor_reports/latest.*` sont restes stale (ex: `running/continue` apres stop manuel ou tick manque), Antidex doit les re-ecrire en etat terminal canonique (`job.status=done|error`, monitor `done|failed` + `wake_developer`) lors d'une simple sync/API read, sans attendre un nouveau tick monitor.
- 2026-03-11: `waiting_job` validity and missing monitor UX: un ancien `job.json` stale `running` ne doit pas valider `developer_status=waiting_job`; il faut une requete pending ou un vrai job encore vivant. Si un job terminal n'a jamais produit `monitor_reports/latest.*`, l'API/UI expose un monitor synthétique minimal dérivé de `result.json` / de l'état terminal.
- 2026-03-11: Developer postconditions: si `dev_ack.json` + `dev_result.*` existent mais que `developer_status` reste `ongoing`/manquant, l'orchestrateur auto-promote vers `ready_for_review` (ou `waiting_job` si evidence long job), et journalise l'auto-correction dans `data/recovery_log.jsonl`.
- 2026-03-12: Long job scope mismatch: une requete long job hors scope (ex: 2p vs 3p attendu) declenche un blocage Manager avec question actionnable et suppression de la requete invalide, plutot qu'un echec dur du tour.
- 2026-03-10: Long jobs: `developer_status=waiting_job` n'est acceptable que pour un lancement protocol-aware (`--script`, `script_path`, ou argv pointant vers `scripts/`). Une commande metier brute hors wrapper doit etre refusee en postcondition.
- 2026-03-10: Long jobs: une nouvelle requete `REQ-*.json` fraiche prime sur tout ancien job mort. L'auto-restart d'un vieux job ne doit pas court-circuiter un nouveau lancement deja demande par le developer.
- 2026-03-10: Semantique pipeline/job: `Stop pipeline` et `Stop long job` sont distincts. Un run `stopped|paused` peut conserver un long job vivant pour observabilite; l'UI/API doivent le montrer explicitement, et `Stop long job` ne doit pas relancer implicitement le pipeline si celui-ci etait deja arrete.
- 2026-03-10: Corrector vs surveillance proactive: ne pas transformer le Correcteur externe en agent qui corrige en parallele. Direction retenue: ajouter un **auditeur externe periodique** read-only, d'abord en mode `passive`, puis `enforcing` via whitelist de signatures a forte confiance; l'auditeur ecrit un rapport et recommande un incident, puis le flux existant Guardian -> incident -> Correcteur externe prend le relais.
- 2026-03-07: Timeouts Developer: sur `turn/hard_timeout`, stopper l'auto-run (`status=stopped`) au lieu de continuer a boucler Manager/Corrector automatiquement.
- 2026-03-07: Reorientation sans "stagnation": au lieu d'une detection heuristique generale, imposer des points de recul binaires:
  - a chaque `REWORK` (manager_review.md doit contenir diagnostic + `Next actions:` et changer manager_instruction.md ou TODO),
  - a chaque incident/Corrector (question `Q-post-incident` + reponse Manager obligatoire avant reprise).
- 2026-03-08: Reorientation outcome-driven: sur les taches `benchmark|gate|tuning|research|ai_baseline_fix|manual_test`, un `REWORK` doit maintenant inclure un `Goal check` structure (but final, preuve qui invalide, type d'echec, decision, niveau d'action). Si le probleme est amont (`upstream_plan_issue`), le Manager doit reordonner/creer une tache TODO amont et Antidex rebascule ensuite vers le premier item TODO non coche. Les developers doivent aussi remonter un signal structure `What this suggests next` dans leurs resultats pour reduire les reworks myopes.
- 2026-03-11: Direction experimentale des long jobs: une requete protocol-aware peut etre syntaxiquement valide mais conceptuellement hors-scope. L'orchestrateur valide maintenant aussi le **mode de preuve** demande par la tache (`2p` vs `3p`, controle `EASY` avant reruns `MEDIUM` quand explicitement exige dans `manager_instruction.md`) avant d'accepter `developer_status=waiting_job` ou de lancer le job.
