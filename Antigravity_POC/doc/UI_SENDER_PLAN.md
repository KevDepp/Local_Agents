# UI Sender Plan (Antigravity_POC)

Status: implemented (see `ui/`). Debug notes + conclusions: `DEBUG_INJECTION_REPORT.md`.

## Context
We need a fast manual testing UI to send prompts to Antigravity through the existing connector stack:
- Antigravity runs `antigravity-connector` on `127.0.0.1:17375`
- `POST /send` triggers CDP injection (text + submit)

The UI must make it easy to:
- send arbitrary prompts quickly
- choose between starting a new thread or continuing the current one
- avoid duplicate sends (requestId / dedupe)
- debug failures (see `/health`, `/diagnostics`, and the last `/send` response)
- preserve UTF-8 accents end-to-end

## Goals
- Provide a local web UI for sending prompts through `http://127.0.0.1:17375/send`.
- Support `newConversation` toggle:
  - `true`: open a new conversation then send
  - `false`: send to the currently active conversation/panel
- Make sends deterministic enough for repeatable tests:
  - generate unique `requestId`
  - optional `verifyNeedle` token
- Make failures obvious and fast to diagnose.

## Non-Goals
- Selecting an arbitrary existing thread by title/ID (not currently stable).
- Reading Antigravity chat contents back into the UI.
- Replacing the file-based protocol in the POC.

## UX / Screen Layout (Minimal)
- Connection status row:
  - `Health` (app/port/pid) from `/api/health`
  - `Diagnostics` summary (count of `antigravity.*`) from `/api/diagnostics`
  - `CDP` reminder banner if `/send` returns a CDP error
- Sender panel:
  - `Prompt` textarea
  - `New thread` checkbox
  - `verifyNeedle` input (optional; recommended for tests)
  - `notify` checkbox (toast in Antigravity)
  - `debug` checkbox (return full result payload)
  - `Send` button
- Results panel:
  - last response JSON (pretty-printed)
  - history list (timestamp, requestId, ok/method, short error)

## Architecture
Use a tiny local Node.js server to avoid browser CORS issues and to control encoding:
- `Express` server listens on `127.0.0.1:<uiPort>` (ex: 17400)
- Serves static UI assets (HTML/JS/CSS)
- Proxies API calls to the connector at `127.0.0.1:17375`

Rationale:
- Browser-to-connector would require CORS headers from the extension.
- The proxy ensures `Content-Type: application/json; charset=utf-8` and UTF-8 bytes (fixes accent corruption).

## API Contract (UI Server)
All endpoints are served by the UI server and forwarded to the connector.

### GET /api/health
- Forwards to `GET http://127.0.0.1:17375/health`
- Returns JSON `{ ok, app, port, pid }`

### GET /api/diagnostics
- Forwards to `GET http://127.0.0.1:17375/diagnostics`
- Returns JSON `{ app, port, commands }` (UI shows counts)

### POST /api/ping
- Forwards to `POST http://127.0.0.1:17375/ping`
- Body: `{ message: string }`
- Expected: toast "Antigravity Connector: <message>"

### POST /api/send
- Forwards to `POST http://127.0.0.1:17375/send`
- Body (subset):
  - `prompt` (string, required)
  - `newConversation` (boolean, default false)
  - `notify` (boolean, default true)
  - `debug` (boolean, default true)
  - `verifyNeedle` (string, optional)
  - `requestId` (string, required; generated by UI)
- Proxy behavior:
  - Always encode JSON as UTF-8 bytes
  - Always set `Content-Type: application/json; charset=utf-8`
- Returns connector response JSON (store in history)

1. Create `Local_Agents/Antigravity_POC/ui/` package
   - `package.json` with `start` script
   - deps: `express` (and optionally `node-fetch` or use native `fetch` in Node 18+)
2. Implement proxy server
   - static files from `ui/public/`
   - `/api/*` endpoints forward to connector
   - short request timeout (ex: 10s) with clear error JSON
3. Implement UI page
   - on load: poll `/api/health` and `/api/diagnostics`
   - send form: builds payload, generates `requestId`
   - history persisted in `localStorage` (small cap, ex: 50 entries)
4. Add a small runbook section in `README.md` (optional but recommended)
   - start Antigravity with `--remote-debugging-port=9000`
   - confirm connector `/health`
   - launch UI and send a smoke test

## Test Plan (UI)
- Manual smoke test:
  - Click `Ping` -> toast appears in Antigravity
  - Send prompt with `New thread = true` and `verifyNeedle = <token>`
  - Success criteria: prompt input is cleared (submitted) and a new thread appears with the prompt
- Manual continue-thread test:
  - Select an existing thread in Antigravity
  - Send with `New thread = false`
  - Success criteria: message appears in the currently active thread
- Encoding test:
  - Prompt containing accents: `éèàôù ç`
  - Success criteria: no `�` replacement characters in the sent message

## Acceptance Criteria
- UI can send prompts reliably without using the CLI scripts.
- `New thread` vs `Continue` behaves as expected (within the limitation that "continue" uses the active thread).
- Accented characters are preserved end-to-end.
- UI surfaces failures with enough detail to debug quickly (health/diagnostics + last response).

## Risks / Limitations
- Continuing a specific thread by ID/title is not implemented; it depends on the currently active thread in Antigravity.
- CDP target selection can still pick a wrong target; the connector should keep improving target detection.
- Antigravity UI updates may change DOM selectors (Lexical/editor/button heuristics).

## Implementation Notes (To Make It "Perfect")

These are practical gotchas from the recent debugging sessions.

### Do Not "Fix" The Extension By Forcing Defaults
- Do not change `activationEvents` to `"*"`: the connector already activates (`onStartupFinished`) when correctly installed.
- Do not hardcode `useCDP=true` in the extension manifest: keep CDP as an explicit user/workspace setting.
- The UI sender should work with the current contract: connector on `127.0.0.1:17375`, CDP enabled via settings, Antigravity started with remote debugging when needed.

### Preconditions Checklist (UI Should Surface This)
- Connector reachable: `GET /api/health` returns `{ ok:true, app:"Antigravity", port:17375 }`.
- CDP available when `useCDP=true`: `http://127.0.0.1:9000/json/version` reachable (Antigravity launched with `--remote-debugging-port=9000`).
- If either precondition fails, disable the Send button and show a clear instruction.

### Encoding: Always Send UTF-8 Bytes
We repeatedly observed `�` replacement characters when requests were not treated as UTF-8 end-to-end.
- Proxy must forward request bodies as UTF-8 bytes (not as a JS string).
- Always set `Content-Type: application/json; charset=utf-8` on the proxy->connector request.
- Recommended Node pattern:
- `const bytes = Buffer.from(JSON.stringify(payload), "utf8");`
- `fetch(url, { method:"POST", headers:{ "content-type":"application/json; charset=utf-8" }, body: bytes })`

### Dedupe / Request IDs (Avoid Double Sends)
The connector dedupes by `requestId`/`runId` for ~60 seconds.
- UI must generate a unique `requestId` per click (timestamp + random suffix).
- On retries, re-use the same `requestId` (explicit "Retry" action) to avoid accidental duplicates.
- UI should show the last `requestId` next to the response for traceability.

### Verification: Use An ASCII Token
CDP verification is easier and less brittle with a short ASCII token.
- Default `verifyNeedle` should be ASCII, e.g. `AG_UI_YYYYMMDD_HHMMSS_xxxx`.
- UI can offer a button "Insert token into prompt" that prepends a line like:
- `TOKEN: AG_UI_...`
- Then set `verifyNeedle` to that token.

### New Thread vs Continue Semantics
At this stage, "continue" means "send to the currently active conversation/panel in Antigravity".
- `New thread = true` should send `{ newConversation: true }`.
- `Continue = true` should omit `newConversation` (or send `false`).
- Do not attempt to select a specific thread by title/ID yet (it is not stable).

### Debugging Controls (High Value)
UI should include:
- A `Ping` button (calls `/api/ping`) to confirm we are talking to the intended Antigravity window.
- A `Debug` toggle (send `debug:true`) and display full JSON response.
- A "Copy curl" button that prints an equivalent request (useful for repro).

## Run (One Command)
From `Local_Agents/Antigravity_POC/`:
- `npm run ui`

This starts the UI server in the background, waits until it responds on `http://127.0.0.1:17400`, then opens the browser automatically.
